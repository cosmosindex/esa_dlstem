# MOT 系统性对比框架:TBD vs JDT,按物体尺寸分析

> 跨范式比较 TBD(SORT / ByteTrack / OC-SORT / BoT-SORT / TrackTrack)与
> JDT(FairMOT / TGraM)的方法论。核心:MOT 任务只有两轴 —— **检测能力**与
> **关联能力**,各自 **vs 物体像素尺寸**分析,寻找小物体处的断崖。

---

## 0. 出发点

MOT 任务无非比较两点:

1. **检测能力**:物体变小,模型对小物体的检测能力是否显著下降(预期有断崖)。
2. **关联能力**:在不同大小物体下,模型能否把检测和预测正确关联起来。

要做的就是把这两点**干净地、公平地隔离出来**,按尺寸分 bin 观察。**收敛成两个实验,
不要再往里加 lane。**

---

## 1. 两个实验

### 实验一 —— 检测能力 vs 尺寸

- 拿每个方法**自己的检测框**和 GT 比:
  - JDT 解码 `hm/wh/reg` 出框;
  - TBD 用它的检测器(HiEUM 车 / FasterRCNN 其他类)的框。
- **按物体像素大小分 bin**。
- 指标:**Recall / Precision / AP** per bin。
- 产出:检测指标 vs 尺寸曲线,一条线一个方法 → 暴露**小物体检测断崖**。

> 这里没有"公平"问题 —— 用谁的检测器,就是这个方法的检测能力,这是真实属性。

### 实验二 —— 关联能力 vs 尺寸(从检测里隔离)

- **同一份 GT 框喂给每一个方法**(TBD 和 JDT 都用)。
- 各方法跑**自己的关联**(运动 / 外观)。
- **按尺寸分 bin**。
- 指标:**AssA / IDF1 / IDsw** per bin。
- 因为检测输入对所有人完全相同,**任何差异都纯粹是关联能力**(包括 JDT 弱 ReID 的拖累)。

> **GT 框 = 关联轴的"均衡器"**:把检测影响清零,只剩关联。

---

## 2. 关键认知:GT 框对弱表征 JDT 也是**公平的**

> 顾虑:"把 GT 框塞进一个表征太弱、分不出小物体的 JDT,这不还是不公平吗?"
>
> **不。这恰恰是正确、公平的测量,而且这就是结论本身。**

- **公平 = 在被测能力的边界处,输入完全相同**(所有人都拿同一份 GT 框)。
  公平**不等于**"补偿 JDT 的弱点"。
- JDT 拿到完美的框,但它的 stride-4 ReID embedding 分不出小物体 → AssA 低。
  这个低分是 JDT 的**真实关联能力**;原因(表征弱)正是你想证明的论点。
- 不要把"**公平对待**"(fair)和"**讨好/补偿**"(flattering)搞混。**证明
  "JDT 输在表征"的实验,正是这个 GT 框关联测试 —— 低分是证据,不是 artifact。**

---

## 3. JDT 怎么做 GT-box oracle(它能做,但要动手术)

JDT 没有外部"框输入口":它的 ReID 是从 `id` 特征图上、**在检测解码出的中心点索引处**
采样得到的。见 `TGraM/src/lib/tracker/multitracker.py:256-258`:

```python
dets, inds = mot_decode(hm, wh, reg=reg, ...)            # 从热力图解码中心 + 扁平索引
id_feature = _tranpose_and_gather_feat(id_feature, inds) # 在检测中心处采样 ReID 向量
```

**Oracle = 只替换这一行**:

```python
inds  = gt_center_indices(gt_boxes)   # GT 框中心映到 /4 网格: floor(cy/4)*W + floor(cx/4)
dets  = gt_boxes_with_score_1(gt_boxes)
id_feature = _tranpose_and_gather_feat(id_feature, inds)  # 仍采样"模型学到的"ReID,位置换成 GT 中心
# 下游 Kalman + 匈牙利匹配 完全不动
```

约 30–50 行包一层 `JDETracker`;FairMOT 与 TGraM 共用同一补丁。

> **stride-4 硬天花板**:相距 <4 像素的两个 GT 物体,`floor(cx/4)` 落进**同一格 →
> 同一个 `ind` → 共享同一个 embedding → 架构上无法区分**。GT 框测试正好把这个底逼出来。

---

## 4. 可选深挖(主对比**不需要**)

若实验二里 JDT 关联分低,想进一步区分是"**匹配算法差**"还是"**特征差**":
把**同一个特征**喂给所有方法的关联器。

- 换了特征 JDT 追平 → **特征(表征)问题**。
- 追不上 → **算法问题**。

这只是给"为什么"补一个注脚,**不是主 lane**。

---

## 5. 元原则(根治"对比永远差一点公平"的焦虑)

**跨范式拿不到一个单一的公平数;你拿到的是一个分解,每条轴明确声明"固定了什么、放开了什么"。**

- 检测公平不了 → **GT 框把检测器拿掉**。
- 训练 / ReID 数据不对称(JDT 域内训练、SORT 无训练、BoT-SORT-ReID 用 MOT17)
  是**范式定义层面的真实属性** → 当成一列 "**ReID 训练数据**" 声明出来,别隐藏、别强行对齐。
- **混淆变量 → 当成结果列出来,不是方法漏洞。**

实操要点:

- **bin 切法**:照 `docs/bbox_stats/` 里的真实尺寸分布按分位数切,**别用 COCO 的
  32²/96²**(对卫星小物体太粗,全挤在 "small")。
- **HOTA 的 `DetA × AssA` 分解**就是这两轴的原生工具:DetA 装检测,AssA 装关联,
  永远同时列。

---

## 一句话总结

| 轴 | 实验 | 输入 | 指标 |
|---|---|---|---|
| 检测能力 | 各方法自己的框 vs GT,按尺寸分 bin | 原始视频 | Recall / Precision / AP |
| 关联能力 | **GT 框喂所有人**,各自关联,按尺寸分 bin | **同一份 GT 框** | AssA / IDF1 / IDsw |

JDT 在关联轴上即使拿到 GT 框还是低 → **这就是"它表征弱"的证明,是结果,不是不公平。**
GT 框对所有人一视同仁,谁弱谁就低 —— 这正是你想看到的。
